# Benchmark Construction（需求与方法论 · 去实现细节版）

> 本节从"我们想要什么、我们怎么做"的角度描述 code-bench,不涉及任何实现细节(文件、函数、数据结构)。

## 0. 这一节在做什么

code-bench 把**一个真实代码仓库**自动转化为一套**黑盒重建考卷**:给候选一份"对外行为规格",让候选**从零重建**一个行为等价的实现,再用一套**隐藏的判卷集**自动评分。所有"针对某个具体仓库的创造性工作"都交给一个无头 agent 在运行时完成,框架本身不为任何具体仓库写死逻辑。

我们对这套构建过程提出两类东西:一组**设计需求**(构建必须满足的不变量),和一套**构建方法论**(把原仓变成考卷的步骤)。

## 1. 设计需求（构建必须满足的不变量）

- **R1 仓库无关、由 agent 驱动**:框架不内置任何程序专属逻辑;每个仓库的"事实"由人最小化地登记,"理解/契约/能力划分/测试设计"等创造性工作由 agent 在运行时产生。新增一个仓库不应需要改框架。
- **R2 差分 oracle(真值取自参考实现)**:期望输出**从不由人或 agent 编写或猜测**;它来自实际运行原仓(参考实现)并冻结其真实行为。出题者只决定"喂什么输入、看哪些输出",不决定"输出应等于什么"。
- **R3 确定性**:只有在参考实现上**可复现**的行为才被冻结为真值(对同一输入多次运行结果须一致),非确定的输入被剔除;有状态系统在每个用例前清空状态,出题与判卷用同一套状态清理策略。
- **R4 跨实现公平**:对每个被观测的输出,按"最宽松但仍能锁住正确行为"的强度比较;版本号、时间戳、顺序、错误文案等**不同正确实现本就会不同**的表面,不做严格(逐字节)比较。
- **R5 纯黑盒、只在边界判卷**:只在进程/网络/文件边界观测候选行为,**从不读候选内部结构或源码逻辑**来判分。
- **R6 质量优先于配额**:每个能力要求达到一个最低测试量,但**质量门高于配额**——凑不够合格样本时显式标记待复核,绝不用低质量样本填充。
- **R7 出厂前可靠性验证**:卷子发布前必须通过两道关——参考实现自身应近乎全部通过(否则卷子有误),且卷子能**识别出错误实现**(注入故障可被杀)。
- **R8 失败必须显形**:任何异常都不会被静默跳过并当作通过;判定规则是机械、可复算、可审计的。
- **R9 反记忆、反作弊**:真值与判卷集对候选隐藏;候选只凭规格作答;支持**跨语言重建**以压制记忆;不向候选泄露参考实现的身份。
- **R10 题面与判卷分离**:产物明确分成"给候选的规格"与"隐藏的判卷/验卷",这条可见/隐藏边界既是任务定义,也是反作弊的基础。

## 2. 构建方法论（从原仓到考卷）

我们把一个仓库按以下步骤反向构建成考卷;每一步都是"为达成某个需求而做的一件事",不绑定任何实现:

1. **获取可运行的参考实现,并确认其行为可观测。** 把原仓构建成一个可运行的参考实现,确认它的行为完全暴露在命令/网络/文件边界上;不满足(如仅靠图形界面才有输出)则不纳入。
2. **从原仓提取面向用户的对外 API 使用手册(锁定契约)。** 抽取用户可见的全部对外接口——命令、参数、退出码、输入/输出格式(或 HTTP 端点,或输入→输出规格),形成一份**锁定的契约**;这是候选必须精确实现、判卷也据此绑定的对象。
3. **刻画程序背景与确定性风险。** 概括程序用途与对外可见面;识别外部依赖并区分"可控 / 不可控";标注非确定性来源;判断它现实上**可用哪些语言重写**。
4. **把契约组织成用户可达的能力,并给出带示例的用户故事。** 将零散接口聚成若干"用户能完成的功能",保证每个接口都归属于某个能力(无孤立接口);为每个能力提供**只用契约、不含期望输出**的可执行使用示例。
5. **梳理并可操作化非功能需求(NFR)。** 把性能效率、可移植性等质量要求落成"可在运行边界或源码上观测的探针";黑盒与源码都测不了的指标,显式标注为不可观测并说明原因。
6. **用差分 oracle 合成行为测试并冻结真实输出。** 为每个能力合成覆盖等价类/边界/错误路径的输入;把输入喂给参考实现、**捕获其真实输出作为真值**;只保留在参考实现上可复现的;每个能力达到质量配额,否则标记待复核。
7. **跨实现公平地选择断言强度。** 对每个被观测字段选择"最宽松但仍能锁住行为"的比较强度,避免把实现相关表面(版本/时间/顺序/文案)误判为必须逐字节一致。
8. **对抗式验卷。** 由独立视角审查:是否有过度约束的断言(会冤枉正确但不同的实现)、是否冻进了原仓的偶发/错误行为、是否泄露了白盒内部、NFR 是否对某语言不公平;存在严重问题则不放行。
9. **出厂前验证卷子可靠。** 用参考实现自测整卷(应近乎全过,否则说明卷子有问题);并以"注入错误能否被杀"衡量卷子的**区分度**。
10. **打包成候选题面与隐藏判卷集。** 把规格(契约 + 能力 + 示例 + NFR)交给候选;把行为真值、判卷集与验卷结论隐藏;两侧物理分离。

## 3. 预期产物（从需求角度,而非文件清单）

构建过程对每个仓库产出一份**考卷**,由两半组成:

- **候选可见的"规格"**:程序背景与确定性提示、**锁定的对外契约**、按能力组织的功能说明与使用示例、需满足的非功能需求。它只描述"要实现什么行为",**不含任何期望输出**。
- **隐藏的"判卷与验卷"**:从参考实现冻结的**行为真值与测试集**(差分 oracle)、出厂前的可靠性与区分度结论。候选全程看不到。

数据集层面,产出对每个仓库"构建到哪一步、各项质量指标如何"的汇总,以支撑筛选与复核。

**核心二分**:可见的"规格" vs 隐藏的"真值/判卷"。这条边界一方面**定义了任务**(候选只凭对外行为规格重建),另一方面是**反记忆/反作弊**的依据(真值不可见、跨语言、不泄露参考实现身份)。

## 4. 一句话方法论

> **从一个真实程序反向提炼出"对外行为契约 + 能力示例 + 非功能要求"作为题面,用差分 oracle(运行参考实现冻结真值)构造跨实现公平、确定性、可验证的隐藏判卷集,在严格的质量与出厂门下,产出一套"从规格重建可替换实现"的黑盒考卷。**
