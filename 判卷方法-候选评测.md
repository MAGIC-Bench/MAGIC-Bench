# 候选评测(判卷)方法框架

> code-bench-v2 判卷侧方法学。对应代码:`engine/_grade.py`、`engine/classify.py`、`engine/nfr_score.py`、`engine/replay.py`、`run/grade_worker.sh`、`run/fix_launch.py`、`run/setup_examinee.sh`。

---

## 0. 总览:从候选提交到分数

```
卷面(只给对外契约,看不到原仓源码)
  ↓ 5 个 agent 各用【目标语言】从零重写,产出 源码 + build.sh + run.json(启动命令)
判卷节点:
  ① 构建 build.sh → 二进制
  ② 跑【冒烟suite】→ build_ok(硬门:不过则功能分+运行类NFR全0)
  ③ 跑【功能用例】→ 差分对比原仓golden → 功能分
  ④ 跑【运行时NFR模式】+ 【静态NFR】(codex读源码) → NFR六维
  ↓
score.json = { build_ok, 功能分, nfr_by_dimension{6维} }
```

判分是**黑盒差分**:判卷端不知道"正确实现",只把候选输出和出题时冻结的原仓 golden 按四类断言比对。

---

## 1. 候选要交什么

候选(做题 agent)读卷面后,在自己的工作目录交出:
- **源码**(目标语言的重写);
- **`build.sh`**:在本目录构建出可执行;
- **`run.json`**:`launch` = 启动命令(如 `["./app"]` 或 `["python","main.py"]`)。

判卷端不重新出题,只用出题时打好的 grader(`07_exam/grader/`:`cases/*.json` 功能用例 + `_grader_meta.json` + `nfr_applicable.json` 适用指标)。

---

## 2. 隔离与防作弊(`setup_examinee.sh` / `exam_worker.sh`)

- **非 root 考生**:做题 agent 以 `examinee`(uid 1001,无 sudo)身份跑;golden 根目录 `out/` 被 `chmod 700` root-only,NFS 无 root_squash → examinee **读不到任何 golden / 标准答案**(实测 Permission denied)。
- **无跨场记忆**:每场考试拷一份**全新 HOME**,只植入鉴权(不含历史/会话)→ 天然无跨题数据污染。
- **launch 路径重写**(`fix_launch.py`):做题节点把 launch 写成自己本地绝对路径,判卷在另一节点重建 → 把那段本地路径重写成本机 work 目录(否则候选跑不起来)。
- 判卷以 root 跑,不受 golden 锁影响。

---

## 3. 判卷主流程(`_grade.py main()`)

```python
scen = scenario_type            # cli | service
results = run_service_cases()   if scen=="service"   else  [run_one(c) for c in CASES]
smoke   = [r for r in results if r.smoke]
build_ok = (有smoke and 全部smoke过) or (没有smoke)        # ← 构建门
report = { build_ok, cases:results, modes: run_modes() if build_ok else {} }
statics = static_scores() if build_ok else {}             # codex 静态分(带缓存)
score   = nfr_score.compute_scores(report, 适用指标, statics)
写 score.json
```

---

## 4. 构建门 build_ok(硬门)

- **build_ok = 冒烟suite 全过**(`--help/--version/health` 等活性用例)。冒烟的 golden 是出题时**跨实现公平**冻结的(只在原仓 exit 0 时才钉 exit==0,否则只要有输出)。
- **build_ok=false → 功能分 + 所有运行类 NFR 全部归零**(`nfr_score` 的 build 门)。这保证"建不起来/跑不起来"的卷直接 0,不混入部分分。
- 服务场景:服务起不来 → 所有用例 crashed → build_ok=false。

> 局限:冒烟门偏严——个别题把 `--version`、help 走 stdout 还是 stderr 当冒烟项,缺这种小特性的合理重写会被判 build_ok=false 而全盘 0(约 10 个仓属此类,本数据集保留现状)。

---

## 5. 功能分(差分判分)

`run_one(case)`(cli)逐用例:
1. 建一个全新临时目录,写入用例的输入文件;
2. 跑候选二进制(argv + stdin,cwd=临时目录),抓 **exit / stdout / stderr / 新产生的输出文件**(还顺带量延迟、峰值内存、是否 crash/超时/OOM);
3. **`classify.check(观测, 冻结的断言)`** 逐字段判:
   - exact→字节相等 · normalized→归一化后相等 · invariant→谓词成立 · ignored→恒过;
   - 任一字段不符 → 该用例 fail。

**功能分 = 通过用例数 / 总用例数(0~1)**。
service 场景走 `run_service_cases`:起一次候选服务 + 依赖 DB,**每条用例前重置 DB**,回放 HTTP 步骤后同样按断言判。

---

## 6. NFR 六维评分

适用指标在出题时定(`nfr_applicable.json`),每个指标 0/1,两种打分:

**静态(codex 读候选源码,`static_scores`,带缓存到 `static.json`)**——代码质量/结构/文档/安全:
- MTN1-5(无超大单文件/无循环依赖/无跨层反向依赖/认知复杂度/README完整)、PTB2-6(无硬编码路径/日志不写源码区/显式编码/无环境变量强依赖/无平台强绑定)、SEC2(无硬编码密钥)、CMP1(无硬编码端口路径)。

**运行时(真跑候选,`run_modes` / cases)**——构建/性能/可靠性:
- PTB1(标准构建)、PERF1-4(无崩溃OOM超时 / 尾延迟稳定 / 内存峰值 / 限时通过率≥70%)、
- RLY1(重复跑3轮全过不崩)、RLY5(并发跑不死锁/崩溃)、RLY4(限内存256MB优雅降级)、CMP2(契约 schema 合规);
- service 专属:RLY2(脏重启)、RLY6(强杀重启)、RLY3(断 DB 优雅报错)。

**构建门**:build_ok=false → 所有 NFR 指标强制 0;build_ok=true → 逐指标按上面打。

## 7. 分数结构(`score.json`)

```json
{ "build_ok": true,
  "功能分": 0.4394,
  "nfr_by_dimension": {
    "CMP": {"CMP1":0,"CMP2":null}, "MTN": {"MTN1":0,...,"MTN5":1},
    "PERF": {...}, "PTB": {...}, "RLY": {...}, "SEC": {"SEC2":1} } }
```
`null` = 该指标对这道题不适用。报告里的"满足率"是微观聚合(所有 1 / 所有非null,分母≈题数×每题指标数)。

---

## 8. 判卷基础设施(`grade_worker.sh`)

- **工作队列 + 原子认领**:多节点轮询 `submissions/*/*/SUBMITTED`,`mkdir grade_claims/<id>/<agent>` 原子认领防重判;`GRADE_SKIP` 文件可按 agent 跳过(如 agy)。
- **每次判卷用独立 grader 副本**(`gtmp = mktemp -d`,拷 grader 进去),避免共享 `score.json` 竞争;**热覆盖最新引擎**(把最新 `_grade.py`/`nfr_score.py` 拷进副本)→ 改判分逻辑即时对所有卷生效,不必重新出题。
- **重判**:删 `GRADED` 标记即触发重判;静态分有缓存(`static.json`),重判零 codex 开销。
- **判卷 agent**:静态打分用 **codex**(`model_reasoning_effort=low`、`--sandbox danger-full-access` 只读源码)。

## 9. 已知判卷 bug 与局限(诚实标注)

1. **判卷 bug ①(launch 路径,已修)**:WORKROOT 改 `/tmp/exam_work` 后 `fix_launch` 正则没跟上 → 候选构建成功却跑不起来 → build_ok=false → 判 0。修复后大量"构建失败"翻案(cursor 一度看着垫底,真实是并列第一)。
2. **判卷 bug ②(cwd 竞态,已修)**:grade_worker 删临时目录前没 cd 出去,下一轮 grade.py 在已删目录里启动偶发崩溃 → 兜底判 0。
3. **smoke 门偏严**:见 §4,约 10 个仓因缺 --version 等小特性全员 0,保留现状。
4. **不校验候选语言**:判卷只看对外行为,不查候选用的什么语言 → 16 份提交直接 `git clone` 原仓(写原语言)钻空子,因原仓本就能跑还拿了功能分(cursor 11 份,4 个满分卷是克隆注水)。这是黑盒差分基准的可钻空子,本数据集如实标注。
5. **service 判卷路径未端到端实测**:`run_service_cases` 有"每条用例前重置 DB"的设计,但因数据集 0 道 service(出题侧栽在确定性门,见出题方法 §8),这条路径从未被真实跑过。
6. **数据集 81 道可判**:dosisod-refurb 的 grader 缺失,无法判卷。

---

**一句话**:判卷=黑盒差分。候选只交"对外行为",判卷在隔离环境里构建它、用冒烟门确认它能跑、用四类断言把它的输出与原仓 golden 逐字段比对得功能分、再用静态(codex读码)+运行时(真跑)给 NFR 六维打分;构建不起来则一切归零。
