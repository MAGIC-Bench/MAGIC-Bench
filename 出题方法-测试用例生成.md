# 测试用例生成方法(差分预言机 + 四类断言 + 覆盖率驱动)

> code-bench-v2 出题流水线 stage5 的方法学。所有规则对应代码:`engine/classify.py`、`stages/stage5_loop.py`、`engine/gates.py`、`orchestrate.py`、`agent/agent_stages.py`。

## 0. 总览:出题流水线 stage0-8

每个阶段产出一个工件,先 **JSON Schema 校验** 再过 **业务门**(`gates.py`),不过就停下这道题并记原因;可断点续跑(`STATUS.json`)。编排见 `orchestrate.py`。

| 阶段 | 产物 | 谁生成 | 门(关键规则) |
|---|---|---|---|

| 1 repo-model | `01_repo-model.json`(SUT 类型、对外接口面、候选简介 candidate_brief) | 模型 | scenario_type∈cli\|service\|pipeline;接口面非空;candidate_brief 非空且**不泄露原仓名** |

本文聚焦 **stage5(测试用例生成)**,前置依赖 stage3 的"模块"。

---

## 1. 核心范式:差分预言机(原仓 = 标准答案)

**没有人或模型编造"期望输出"。** 分工是:

- **模型只出"题面"**:每条用例的【输入】+ 每个输出字段的【断言形状】(检查哪个字段、用哪种比对方式),**不含具体期望值**。
- **原仓出"标准答案"**:把输入喂给【原始仓库】跑一遍,原仓的真实输出**就是** golden(`classify.freeze_golden`)。
- **判分**:同一输入喂候选,与原仓冻结的 golden 按断言比对(`classify.check`)。

**为什么这么设计**:期望值由原仓客观产生,杜绝出题者的偏见与笔误;且能低成本批量生成(模型只需想"输入",答案自动来)。

---

## 2. 组织单位:模块与用户故事(stage3)

测试不是散的,是挂在**模块**上的。stage3 由模型把原仓拆成**面向用户的能力模块**,`gate_stage3` 硬卡:

- 每个模块必须 **user_value 非空**(必须是用户能感知的能力,不是内部函数);
- 每个模块 **至少 1 条用户故事**;
- 每条故事 = **prose 四要素**(actor 角色 / precondition 前置 / trigger 触发 / expected 预期)+ **非空可执行代码块**。

模块 id(如 M1/M2…)是 stage5 配额的对象——**每个模块都要被测够**。

---

## 3. 一条用例是怎么生成的(差分预言机的作用)

draft(模型起草,**无期望值**):
```json
{ "id": "...", "modules": ["M1"], "kind": "intent|incident",
  "input": { "argv": ["-m"], "stdin": "<utf8>"  | "stdin_b64": "...",
             "files": {"in.json": {"text": "..."} } },
  "assert_spec": [ {"field":"exit","class":"exact"},
                   {"field":"stdout","class":"normalized","rule":"crlf_lf"} ] }
```

处理流程(`stage5_loop.run` + `classify`):
1. **跑原仓 run1**(允许联网)→ Observation(exit/stdout/stderr/输出文件/http 步骤)。
2. **`freeze_golden(obs, assert_spec)` —— 这就是"差分预言机"这一步,它把 draft 变成真用例**:
   - `exact` → 把原仓输出的**字节值**存进断言;
   - `normalized` → 存**归一化后的值**;
   - `invariant` → 校验该谓词在原仓**确实成立**(不成立 = 模型断言选错 → 丢弃该 draft)。
   - 冻结后 draft 才成为完整用例 `{输入, 带 golden 值的断言}`,写进 `05_tests/<id>.json`。
3. **自检**:用刚冻结的断言回判原仓自己,必须全过(否则引擎 bug)。

**跑原仓这一下"一鱼三吃",直接驱动后续生成:**

| 产出 | 用途 |
|---|---|
| **golden** | 标准答案(上面) |
| **代码覆盖率** `obs.cover_dir`(go build -cover) | 未覆盖代码块 → `hints` → 下一轮喂给模型 → 专门生成"打没覆盖路径"的输入 → 把配额填满 |
| **确定性** | 每个输入**跑两遍**,`classify.agrees` 不一致就丢 → 保证用例可复现 |

→ 所以差分预言机不是"跑完就没用的孤立步骤",而是**每轮迭代的引擎**:起草 → 跑原仓(=golden+覆盖率+确定性)→ 落用例+更新覆盖 → 哪个模块没满配额,就拿覆盖提示再起草 → 循环。

---

## 4. 四类断言(吸收"输出有噪声/非确定"的关键)

不是一律要求字节相同;每个输出字段按其性质归类,见 `classify.py`:

| 类 | 含义 | 可用规则 |
|---|---|---|
| **exact** | 字节完全一致 | (无规则,确定性输出) |
| **normalized** | 经规则归一化后相等 | `crlf_lf`(换行)、`strip`、`rstrip_eol`、`lines_sorted`(行排序,治"对象迭代顺序不定")、`json_canonical`(JSON 规范化)、`regex_extract:<re>` |
| **invariant** | 值无关的谓词成立 | `nonempty`、`empty`、`valid_json`、`regex:<re>`、`eq_int:<n>` |
| **ignored** | 有这个字段但不检查 | (如错误信息的具体措辞、易变的时间戳) |

典型组合:成功用例 = `exit:exact + stdout:normalized(crlf_lf)`;错误用例 = `exit:exact + stderr:invariant(nonempty) + stderr:ignored`(退出码精确、报错非空、但措辞不究)。字段也支持 service 的 `http:<N>:status|body|header`。

## 5. 配额填充循环(每模块刷满 20)

`stage5_loop.run(repo_out, config, draft_provider, quota=20, max_rounds=3)`:

```
per_module = {每个模块: 0}
for 轮 in range(3):
    目标模块 = (第0轮: 全部) | (后续轮: 只挑 还没满20 且 未needs_review 的)
    drafts, 新needs_review = draft_provider(目标模块, 20, 覆盖hints)   # 模型按目标+覆盖提示起草
    for 每个 draft:
        - 没挂模块            → untagged++,丢          # 每条用例必须挂≥1模块
        - argv[0]==程序名      → 丢(req6:程序名不能当 positional 参数传)
        - 签名(prefix+输入)重复 → 合并模块标签(一条用例可同时算多个模块的配额)
        - 跑原仓 run1 → freeze_golden:
            断言非法/invariant在原仓不成立 → 丢(dropped_bad_spec)
            OSError(后端/运行时炸)         → 丢(dropped_run_error)
            超时/exit124(网络挂等环境噪声)  → 丢(dropped_timeout)
        - 跑原仓 run2:两遍 agrees 不一致     → 丢(dropped_nondeterministic)
        - 自检过 → 落用例,per_module[各标签]++ 
    用本轮覆盖更新 hints
    若所有模块都≥20 → break
    若本轮一条没新增  → 把剩下没满的标 needs_review,break
循环后仍<20的模块 → 全部标 needs_review(交人工裁定,不硬废)
```

**"每模块必须有用例"的精确版**:每个模块**要么刷满 20 条、要么被标 needs_review**。`gate_stage5` 卡三条:① 有未挂模块的用例 → 废;② 有模块没到配额且没标 needs_review → 废;③ **<50% 用例钉住内容字段(stdout/stderr/文件)的实值检查**(exact/normalized/regex/eq_int/valid_json)→ 废(光查 exit 太弱,一个 stdout 全是垃圾但退出码对的实现也能蒙混)。

**6 类丢弃**(全记进 `05_coverage-ledger.json`,可审计):未挂模块 / 程序名当参数 / 断言非法 / 运行错 / 超时 / 非确定性。

**intent vs incident**:`intent` = 取自原仓自带 testdata(代表"原仓的本意行为");`incident` = 模型生成的边界/错误用例。

## 6. 两套独立用例

- **冒烟 suite(req 2.8,`05_smoke/`)**:每个项目必有的"活性"用例——cli 用 `--help/--version/-h` + 配置里的 smoke argv;service 打 health 端点;pipeline 跑空输入。也**双跑冻结**,但 golden **跨实现公平**:只在原仓 `exit==0` 时才钉 `exit==0`,否则只要求"有输出"(因为正确的重写可能用不同的 --help 退出约定)。**判卷的硬门(build_ok)就是它**——候选能构建 + 能跑起来。
- **安全用例(req 2.7)**:挂 `security_metric` 标签,是 exit-only 的拒绝/越权探针,**不计入**内容覆盖率分母(否则会机械稀释)。

---

## 7. 三道质量门(stage5/6/7)

- **gate_stage5**(覆盖充分):每用例挂≥1模块;每模块满配额或 needs_review;内容字段实值覆盖≥50%。
- **gate_stage6**(对抗加固):模型做独立对抗审查 → 修复 → 再审,循环到 **open_critical=0**;外加机械检查"断言用了非法 class/rule 关键字"。审查抓:过度约束的 exact、把原仓 bug 当契约冻结、白盒泄露、程序名当 argv 等。**反作弊**:修复不能把断言全改成 `ignored` —— 内容覆盖必须仍≥50%。
- **gate_stage7**(出卷健全):用**原仓重跑自己的 golden**,丢掉"原仓自己都过不了"的不稳用例;**丢弃率>5% 或无幸存用例 → 整道判废**(说明原仓产不出可复现的干净 golden)。

---

## 8. 多 SUT 抽象 + service 为何全部落选

`engine/harness` + `replay.get_backend` 把被测形态抽象成统一接口,draft 的输入形状随场景变:
- **cli**:argv/stdin/文件 → exit/stdout/stderr/输出文件;
- **service**:http 步骤 → `http:<N>:status/body/header`;
- **pipeline**:输入路径 → 输出路径。

**但清单里 22 个 service 仓 0 个进数据集**——不是被过滤,是**栽在 §3 的确定性双跑 + gate_stage7**:有状态服务第二遍运行带着第一遍的状态残留(库写入/已创建资源/会话)→ 两遍输出不一致 → 用例被 `agrees` 丢、原仓重跑对不上自己 golden → gate7 丢弃率飙升 → 整道判废。根因是流水线**没有"每次重跑前把服务状态快照→还原到初始态"的机制**(`runner._snapshot` 只快照文件、不重置服务状态;判卷侧 `restart` 注释 "keep the data dir (dirty restart)" 故意不清状态)。原则上加状态快照/还原即可避免,但通用化难,未做。

## 9. 判卷如何用这些用例

stage8 把 `05_tests/` + `05_smoke/` 打进 grader。判卷(`engine/_grade.py`):
1. 候选构建 + 跑 smoke → **build_ok**(冒烟过才算构建成功;硬门:不过则功能分与运行类 NFR 全 0);
2. 对每条用例:同输入喂候选 → 观测 → `classify.check(观测, 冻结的断言)` 逐字段判过/不过;
3. **功能分** = 用例通过率(0~1);
4. NFR 六维另算(静态由 codex 读源码、运行时真跑候选)。

**一句话**:模型出题面(输入+断言形状),原仓出标准答案(golden),四类断言吸收非确定性噪声,代码覆盖率驱动把每个功能模块刷满 20 条,三道质量门(覆盖/对抗/健全)保证卷子可判、公平、可复现。
