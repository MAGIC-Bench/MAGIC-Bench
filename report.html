① 方法学

这是什么基准:每道题取一个真实开源仓库(原语言 X),让多个 AI 编程 Agent 用另一种目标语言 Y 从零重写, 仅依据卷面给出的对外可观察行为契约(项目描述 / API 手册 / 用户故事 / 非功能需求 / 机器可读 CLI 契约)实现, 看不到原仓源码。判卷为黑盒差分:把候选与"原仓跑出的 golden"在相同输入下对比。
评分项含义
build_ok候选能否构建 + 冒烟(smoke)跑起来。失败则功能分归零，并单独计入构建失败。
功能分单项目差分测试用例的通过率(0~1):候选对外行为与 golden 一致的比例。报告中的平均功能分只对 build_ok=true 的项目做宏平均，构建失败项目不进入功能分分母。
NFR非功能需求,6 个维度逐指标 0/1(null=该指标不适用)。NFR 满足率只统计 build_ok=true 后实际测得的非 null 指标；build_ok=false 的零填充不进入 NFR 分母。CLI 接口契约完全符合项目作为 CMP 兼容性二值指标并入统计。 CMP 兼容性 · MTN 可维护性 · PERF 性能 · PTB 可移植/构建 · RLY 可靠性 · SEC 安全性。

为什么 SUT 全是 cli、没有 service?(出题了,却栽在"出卷健全性门")

结论(代码溯源):清单里有 22 个 service 仓,但无一进入最终数据集 (0/22)。它们不是被选题时过滤掉的,而是都出了题、却产不出"可复现的 golden", 栽在健全性门 gate_stage7 上。根因:有状态服务在出题流水线里缺少"每次重跑前把状态重置到一致初始态"的机制。
证据链(全部来自本基准代码):
  1. service 被尝试出题、未预先过滤:run/launch_chuti.sh 枚举清单全部 121 仓(含 22 个 service),无 scenario_type/friction 过滤;engine/gates.py · gate_stage1 明确允许 scenario_type ∈ cli|service|pipeline;engine/harness.py 注释"enabling … service/pipeline for the dataset";判卷侧 _grade.py 备有完整的 LocalService + run_service_cases + http: 断言字段。service 是一等公民,本该能测。
  2. 出卷健全性要求"原仓能复现自己的 golden":classify.py · agrees() —— golden 捕获时把原仓同一输入跑两遍,只有两遍一致(确定性)的用例才冻结;gate_stage7 —— 再用原仓重跑自己的 golden,丢弃率 >5%(原仓通不过自己 golden)或无幸存用例即判废。
  3. 有状态服务过不了这两关:服务输出依赖累积状态(数据库行 / 已创建资源 / 会话);第二遍带着第一遍的写入 → 两遍不一致 → 用例被 agrees 丢 + 原仓重跑对不上自己 golden → gate_stage7 丢弃率飙升 → 整道判废、进 SKIP(launch_chuti 注释:"golden 不可靠,原仓做不出干净基准")。
  4. 流水线确实没有"重置状态"环节:engine/runner.py · _snapshot() 只快照工作目录文件(捕获文件型输出),不重置服务持久状态;判卷侧 restart() 注释直白:"keep the data dir (dirty restart)" —— 重启故意保留数据目录、不清状态。全流程没有"快照初始态 → 每次重跑前还原"的机制。
这能避免吗?能,但没做。 原则上只要在每次确定性重跑前把服务状态快照 → 还原到初始态(清库 / 还原数据目录 / 重置已创建资源),有状态服务就能产出可复现 golden、通过 gate_stage7。 难在通用化:每个服务存状态的方式不同(库 / 文件 / 内存),流水线没有为每个 service 仓生成"状态快照+还原"的 RESET 钩子,故有状态服务全部落选。 所以本数据集"全 cli"是这条流水线局限的产物、而非有意排除——harness 本身支持 service,是 golden 生成环节复现不了有状态 SUT。

被判废的 22 个 service 仓按领域:web_api 14 · database_storage 6 · cryptography_security 2
